# Development Guide

> See also: [Architecture](./architecture.md) · [API Reference](./api-reference.md)

---

## Requirements

- Node.js 22.5+
- Yarn 4 (managed via corepack — run `corepack enable` once)
- PostgreSQL 14+ (local or via Docker)

---

## Setup

```bash
corepack enable          # activate Yarn v4 (once per machine)
yarn install             # install backend dependencies

# Start a local postgres (or use your own):
docker run -d --name pg-logger \
  -e POSTGRES_USER=logger -e POSTGRES_PASSWORD=logger -e POSTGRES_DB=logger \
  -p 5432:5432 postgres:17-alpine

yarn dev                 # backend API at http://localhost:3000

# In a second terminal — frontend with hot-reload:
cd frontend
yarn install             # first time only
yarn dev                 # Quasar dev server at http://localhost:9000
```

During development, access the app at **http://localhost:9000** (Quasar dev server). It proxies `/api/*` requests to the backend on port 3000. The backend alone at port 3000 serves the compiled SPA; in dev that directory may be empty, which is fine.

The schema is created automatically on first startup. The watcher will backfill any existing files in `logs/` in the background.

## Yarn Scripts

### Backend (root)

| Script | Command | Description |
|---|---|---|
| `yarn dev` | `tsx watch src/server.ts` | Backend dev server with hot-reload at :3000 |
| `yarn build` | `tsc` | Compile TypeScript to `dist/` |
| `yarn start` | `node dist/server.js` | Run compiled production server |
| `yarn lint` | `eslint src/` | Lint TypeScript source |
| `yarn format` | `prettier --write src/` | Format source files |
| `yarn ingest` | `tsx src/ingest.ts` | Ingest all log files |
| `yarn ingest:force` | `tsx src/ingest.ts -- --force` | Force re-ingest all files |
| `yarn ingest:check` | `tsx src/ingest.ts -- --check` | Show ingestion status |

### Frontend (frontend/)

| Script | Command | Description |
|---|---|---|
| `yarn dev:ui` | `cd frontend && quasar dev` | Quasar dev server with HMR at :9000 |
| `yarn build:ui` | `cd frontend && quasar build` | Build SPA to `frontend/dist/spa/` |

## Environment Variables

| Variable | Default | Description |
|---|---|---|
| `DATABASE_URL` | derived | Full postgres connection string. Takes priority. |
| `POSTGRES_USER` | `logger` | Used if `DATABASE_URL` not set |
| `POSTGRES_PASSWORD` | `logger` | Used if `DATABASE_URL` not set |
| `POSTGRES_HOST` | `localhost` | Used if `DATABASE_URL` not set |
| `POSTGRES_PORT` | `5432` | Used if `DATABASE_URL` not set |
| `POSTGRES_DB` | `logger` | Used if `DATABASE_URL` not set |
| `LOGS_DIR` | `./logs` | Directory containing log files |
| `PORT` | `3000` | HTTP listen port |
| `POLL_MS` | `5000` | Watcher polling interval in ms (0 = disable) |

## TypeScript Notes

- **Backend** (`tsconfig.json`): targets ES2022, outputs CommonJS to `dist/`, `moduleResolution: Node`, strict mode
- **Frontend** (`frontend/tsconfig.json`): extends `@quasar/app-vite` preset, `moduleResolution: bundler`, `noEmit: true` (Vite handles transpilation)
- Run `yarn build` before `yarn start` in production
- `tsx` is used for backend dev/scripts — no compilation needed
- All Vue components use `<script setup lang="ts">` with generic `defineProps<{}>()` / `defineEmits<{}>()`

## Adding a New API Route

1. Create `src/routes/my-route.ts` exporting `async function myRoutes(fastify, opts)`
2. Import `sql` from `../db/index.js` for database access
3. Use `WHERE 1=1 ${fileFilter(q.file)} ...` pattern from `utils.ts` for dynamic filters
4. Register the plugin in `src/routes/index.ts`

---

## Adding a New Frontend Component

1. Create `frontend/src/components/MyComponent.vue` using `<script setup lang="ts">`
2. Import from `../api` for data fetching, `../stores/appStore` for shared state
3. Watch `store.selectedFile` and reload data when it changes
4. Import and use in the relevant tab component (`DashboardTab.vue`, `LogsTab.vue`, or `ChartsTab.vue`)

## Docker

```bash
docker compose up -d         # build + start both postgres and file-logger
docker compose down -v       # remove containers AND pg-data volume (full reset)
```

The `file-logger` service uses `depends_on: condition: service_healthy` — it will not start until `pg_isready` reports healthy on the postgres container.

Build stages (Dockerfile):
1. **Builder (backend)**: `corepack enable` → `yarn install --immutable` → `yarn build` (tsc)
2. **Builder (frontend)**: `yarn install --immutable` in `frontend/` → `quasar build`
3. **Runtime**: copies `node_modules` + `dist/` from backend builder; copies `frontend/dist/spa/` → `src/public/`; `CMD ["node", "dist/server.js"]`
